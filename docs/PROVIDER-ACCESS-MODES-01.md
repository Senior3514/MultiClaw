# MultiClaw Provider Access Modes 01

## Principle

MultiClaw should not assume that every user connects models in only one way.

## Supported direction

Provider access should eventually support multiple modes, including:
- direct API keys
- account-connected or subscription-backed provider access where applicable
- router providers
- local or self-hosted models

## Product meaning

The user should be able to bring model access in the form that best fits their environment.
The product should not force one narrow connection model.

## Standard

MultiClaw should feel provider-flexible, access-flexible, and portable.
